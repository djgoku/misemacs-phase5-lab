#+TITLE: misemacs
#+OPTIONS: toc:nil num:nil

Reproducible, self-contained [[https://www.gnu.org/software/emacs/][GNU Emacs]] builds for Apple-Silicon macOS — compiled hermetically from a conda-forge toolchain, packaged as a relocatable =Emacs.app=, and installable with [[https://mise.jdx.dev/][mise]] + [[https://aquaproj.github.io/][aqua]].

misemacs builds Emacs from [[https://github.com/emacsmirror/emacs][=emacsmirror/emacs=]] on a daily cadence that *only publishes when upstream actually changed*. Adding a new Emacs version is a *data change, never a code change*.

* What you get

- *Self-contained =Emacs.app=* — the whole dylib closure (GnuTLS, libxml2, tree-sitter, …) is bundled under =Contents/Frameworks= and its load paths rewritten to resolve from inside the app. No Homebrew, no pixi, no system libraries needed at runtime; it runs on a clean machine.
- *Reproducible* — every C dependency is pinned by a committed =pixi.lock= (the full transitive closure from conda-forge); the pipeline toolchain is pinned by =mise.lock=.
- *GUI (Cocoa/NS) build* with tree-sitter, libxml2, and GnuTLS. JSON is native (no jansson). native-comp is off in v1.
- *CalVer releases* tagged =emacs-<channel>-YYYY-MM-DD= (e.g. =emacs-master-2026-06-14=, =emacs-31-2026-06-14=), each shipping a =SHASUMS256.txt= and ad-hoc code signatures.
- *Bundled spell-checking* — [[https://github.com/AbiWord/enchant][enchant]] (with the AppleSpell + Hunspell providers) ships *inside* the app, relocated to resolve from within the bundle, so [[https://github.com/minad/jinx][jinx]] compiles and runs on a clean machine with no Homebrew/system enchant. See *Spell-checking* below.
- On your =PATH= after install: =Emacs=, =emacsclient=, =etags=, =ebrowse=, plus =enchant-2= and =enchant-lsmod-2= (the bundled spell-checker CLIs).

*Scope (v1):* macOS =arm64= only, native-comp disabled, ad-hoc signed (not Developer ID / notarized). See the [[docs/superpowers/specs/2026-06-05-bundled-emacs-build-system-design.md][design spec]] §15 for what is designed-for but deferred (native-comp, more arches/OSes, Developer ID, =-nw=).

* Install

misemacs ships its own single-file aqua registry ([[aqua/registry.yaml][=aqua/registry.yaml=]]) with *one package per channel* — =djgoku/misemacs-emacs-master= and =djgoku/misemacs-emacs-31=. Point mise at it, then install the channel you want:

#+begin_src sh
export MISE_AQUA_REGISTRY_URL=https://raw.githubusercontent.com/djgoku/misemacs/main/aqua/registry.yaml

# the most recent master build (swap in -emacs-31 for the Emacs 31 line):
mise use aqua:djgoku/misemacs-emacs-master@latest

# …or pin a specific dated build (the version is the date; the channel is the package):
mise use aqua:djgoku/misemacs-emacs-master@2026-06-14
#+end_src

=Emacs= and friends are then on your mise =PATH=:

#+begin_src sh
mise exec -- Emacs --version
#+end_src

mise verifies the downloaded asset against GitHub's release digest. (The first release appears once the daily workflow has run — see *How it works* below.)

** Per-channel artifact repos

Each channel publishes its releases to its *own* GitHub repo:

| aqua package                      | artifact repo                       |
|-----------------------------------+-------------------------------------|
| =aqua:djgoku/misemacs-emacs-master= | [[https://github.com/djgoku/misemacs-emacs-master][=djgoku/misemacs-emacs-master=]]      |
| =aqua:djgoku/misemacs-emacs-31=     | [[https://github.com/djgoku/misemacs-emacs-31][=djgoku/misemacs-emacs-31=]]          |

The aqua *package names* are unchanged; only the underlying repo each package reads from has changed. The registry itself ([[aqua/registry.yaml][=aqua/registry.yaml=]]) still lives here in =djgoku/misemacs=, and =MISE_AQUA_REGISTRY_URL= continues to point at it.

*Why separate repos?* mise (via aqua) fetches only the first ~100 entries of a repo's tag/release list. If both channels shared a single repo, a high-frequency channel (e.g. =master=, built daily) would eventually push a low-frequency channel's tags off that page, making older — or even all — versions of that channel uninstallable. One repo per channel eliminates the interference.

*=@latest= is marker-independent.* The registry uses =version_source: github_tag=, so =@latest= and =mise up= always resolve to the newest *tag* in the channel's own repo — independent of which release GitHub marks as "Latest". A =@latest= pin rolls automatically with =mise up=; no =--bump= is needed. =--bump= is only required when moving from one exact dated pin to another.

*Fresh-channel caveat.* A newly-added channel has no installable version until its first daily build publishes — its artifact repo starts empty.

* Spell-checking (bundled enchant)

Every =Emacs.app= bundles [[https://github.com/AbiWord/enchant][enchant]] under =Contents/Resources/enchant/= as a self-contained, relocated sub-prefix (its own =libenchant=, the Hunspell + AppleSpell providers, and their dylib closure). enchant self-relocates at load time (a =dladdr= constructor), so it resolves entirely from inside the app — no Homebrew, no system enchant.

** jinx (primary)

[[https://github.com/minad/jinx][jinx]] compiles a small C module against =libenchant= on first load. misemacs ships everything that compile needs — headers, a relocatable =enchant-2.pc=, and a scoped =pkg-config= shim — plus a discovery-only =site-start.el= that points jinx's compile at the bundle. So on a clean machine:

#+begin_src sh
# install jinx as usual (ELPA), then enable it — it compiles against the bundled enchant:
emacs --eval "(progn (package-install 'jinx) (global-jinx-mode))"
#+end_src

jinx still needs the *Xcode Command Line Tools* (=cc=) on the user machine to compile its module — that is jinx's own requirement, not something a bundle can remove. "Clean machine" for jinx therefore means "clean except CLT". On an Emacs upgrade the bundled =site-start.el= repairs jinx's embedded library path in place (no recompile). Opt out entirely with =(setq misemacs-enchant-disable t)= or =emacs -Q=.

** ispell.el (bonus)

The =enchant-2= CLI is on your =PATH=, so ispell can use it directly:

#+begin_src elisp
(setq ispell-program-name "enchant-2")
#+end_src

** Backends — AppleSpell default, Hunspell optional (bring your own dictionary)

The default backend is *AppleSpell* (macOS's built-in, multilingual, license-free spell checker); the bundle ships the AppleSpell locale map so =en_US= / =en_GB= / … check and suggest out of the box with *no dictionary files required* (=enchant-lsmod-2= lists the available providers).

The *Hunspell* provider is also bundled, but *no Hunspell dictionaries ship with the app* — Hunspell is bring-your-own. To use Hunspell for, say, =en_US=:

1. Obtain the two dictionary files =en_US.aff= and =en_US.dic= — e.g. from the [[https://github.com/LibreOffice/dictionaries][LibreOffice dictionaries]] repo (=en/=), your distro's =hunspell-en-us= package, or generate them at [[http://app.aspell.net/create][app.aspell.net/create]].
2. Put both files in =~/.config/enchant/hunspell/= (the per-user directory the Hunspell provider searches — no environment variables needed).
3. Tell enchant to prefer Hunspell — add =en_US:hunspell= to =~/.config/enchant/enchant.ordering=.

* How it works

A small *Elixir orchestrator* makes every decision (pure, unit-tested functions), a *bash pipeline* does the heavy lifting, and *mise + pixi* lock every input. GitHub Actions runs it daily behind a cheap change-detection gate, so the expensive macOS build only fires when upstream moved.

#+begin_example
  decide ─▶ build ─▶ finalize
    │         │         │
    │         │         └─ merge manifest fragments, flip "Latest"
    │         └─ matrix: one cell per (changed version × target):
    │              build → relocate → package → publish
    └─ change gate: nothing changed ⇒ publish nothing
#+end_example

| Stage      | Owner          | What it does                                                                            |
|------------+----------------+-----------------------------------------------------------------------------------------|
| =decide=   | Elixir         | Resolve upstream SHAs, diff against the last manifest, emit the dynamic build matrix     |
| =build=    | bash + pixi    | =./configure= + =make= a GUI Emacs against the locked per-version pixi env               |
| =relocate= | Elixir         | Bundle the dylib closure, rewrite load paths, ad-hoc sign; gated by an =otool= check     |
| =package=  | bash           | xattr-free tarball + =SHASUMS256.txt= + self-verify                                      |
| =publish=  | bash           | Create the GitHub release (asset + checksums)                                           |
| =finalize= | Elixir + bash  | Merge per-cell manifest fragments, attach =build-manifest.json=, flip the "Latest" marker |

The daily workflow ([[.github/workflows/daily.yml][=.github/workflows/daily.yml=]]) runs on a cron, on =workflow_dispatch= (with an optional =force_version=), and on pull requests touching =versions/= or =*.toml= (as a no-publish dry-run).

* Repository layout

#+begin_example
misemacs/
├── versions.toml          # THE manifest: one row per Emacs version (the build list)
├── targets.toml           # OS × arch matrix (v1: macos-arm64 only)
├── versions/
│   ├── master/            # per-version build inputs …
│   │   ├── mise.toml      #   EMACS_REF + configure flags + pixi-env wiring
│   │   ├── pixi.toml      #   conda-forge build deps
│   │   └── pixi.lock      #   full transitive lock (committed)
│   └── emacs-31/          # … a second version = a copied dir + one toml row
├── orchestrator/          # Elixir brain: pure decision fns + Mix tasks (ExUnit-tested)
├── pipeline/              # bash stages: build-emacs, package, publish, promote
├── scripts/               # configure-check, e2e-aqua-install
├── aqua/registry.yaml     # the vendored aqua registry users consume
├── mise.toml              # shared task seam (test/lint/build/decide/finalize/…)
└── .github/workflows/     # daily.yml (the automation) + orchestrator-ci.yml
#+end_example

* Adding a build

A "build" is a *version*: one row in =versions.toml= plus a =versions/<name>/= directory holding =mise.toml=, =pixi.toml=, and =pixi.lock=. Adding a new channel also requires a one-time artifact-repo setup — after that, the daily pipeline builds and publishes it automatically.

=emacs-31= is a worked example. The full steps are:

1. Add a row to [[versions.toml][=versions.toml=]] (=ref = "emacs-31"=, =channel = "31"= → tag =emacs-31-YYYY-MM-DD=).
2. Add a [[versions/emacs-31/][=versions/emacs-31/=]] directory copied from =master/= with =EMACS_REF= flipped to =emacs-31=. The =pixi.lock= is reused verbatim since build dependencies don't depend on the Emacs version.
3. Add a package entry to [[aqua/registry.yaml][=aqua/registry.yaml=]] pointing at =repo_name: misemacs-emacs-<channel>= (copy an existing entry and adjust =name=, =repo_name=, =description=, and =version_prefix=).
4. Create the artifact repo (one-time). It MUST have an initial commit — GitHub returns =HTTP 422 "Repository is empty"= when you try to create the first release/tag on a repo with no default branch, so pass =--add-readme=:
   #+begin_src sh
   gh repo create djgoku/misemacs-emacs-<channel> --public --add-readme
   #+end_src
5. Install the publishing GitHub App on the new artifact repo so the daily workflow can push releases to it.

After step 5, the daily pipeline detects the new version, builds it, and publishes the first release to =djgoku/misemacs-emacs-<channel>= — at which point =mise use aqua:djgoku/misemacs-emacs-<channel>@latest= becomes installable.

The layout contract is enforced by the =version_layout_test= (every =versions.toml= row must have its three input files committed), so a malformed addition fails loudly.

* Development

misemacs uses [[https://mise.jdx.dev/][mise]] as the one task runner across local dev, the pregate hooks, and CI:

#+begin_src sh
mise run test     # orchestrator ExUnit suite (mix test --warnings-as-errors)
mise run lint     # mix format --check-formatted
mise run fmt      # mix format
#+end_src

Heavier build-related tasks (run =mise tasks= for the full list): =build=, =relocate=, =package=, =publish=, =configure-check=, =cleanroom=, =decide=, =finalize=.

Before pushing, the =.pregate/= hooks run the test/lint suite on Linux and a no-pixi cleanroom launch inside a disposable macOS VM. Design specs, per-phase plans, and the validation log live under [[docs/superpowers/][=docs/superpowers/=]].
